# 基于 C 语言的跨平台智能英语单词记忆系统 —— 课程设计报告

---

## 摘要

本文阐述了一款面向英语学习者的桌面级单词记忆软件的设计与实现。随着全球化进程的加速，英语作为国际通用语言的重要性日益凸显，而词汇量积累是英语学习的关键瓶颈。传统纸质单词书存在携带不便、缺乏交互反馈、无法追踪学习进度等缺陷，市面现有的单词 App 则多依赖移动端或 Web 技术栈，启动缓慢且存在隐私隐患。针对上述痛点，本系统采用 **C11 标准**与 **raylib 多媒体图形库**构建了一款高性能桌面端单词记忆工具，通过**即时模式图形用户界面（Immediate Mode GUI）**范式，在 Linux、Windows 及 macOS 平台实现一致的交互体验。

系统核心包含用户账户管理、单词库维护、多模式智能复习、自适应学习计划及跨语言文字渲染五大子系统。设计上遵循**全局单例状态管理**与**函数指针驱动导航**的架构模式，以纯文本管道分隔文件实现轻量级数据持久化，不依赖外部数据库引擎，真正做到零配置、开箱即用。项目采用 CMake 构建系统管理跨平台编译，并通过 GitHub Actions 实现了 Linux AppImage 与 Windows zip 便携版的自动化构建与发布。

**关键词：** 即时模式 GUI；间隔重复；C11；跨平台图形应用；自适应学习系统

## 一、系统总体架构

系统的设计目标是一方面提供丰富的学习功能，另一方面保持极低的部署复杂度——用户下载即用，无需安装数据库、运行时或任何第三方依赖。为实现这两个看似矛盾的目标，系统在架构层面做了多项取舍。

### 1.1 架构概览

系统采用 **分层单向依赖** 架构，自底向上分为数据层、业务逻辑层、UI 基础组件层与页面展示层。每一层只依赖其下层提供的接口，不允许逆向调用。各层之间通过全局单例状态对象 `AppState` 进行数据共享，避免跨层指针传递带来的耦合。这种架构的好处是每层可以独立测试和替换：例如，将数据层从文本文件切换为 SQLite 数据库时，只需修改业务逻辑层中的文件读写函数，UI 层完全不受影响。

```
┌─────────────────────────────────────────────────────────────┐
│                     页面展示层                              │
│  主页 │ 浏览学习 │ 卡片背单词 │ 选词背单词 │ 测试 │ 搜索   │
│       学习计划 │ 设置 │ 账号管理 │ 词库管理                 │
├─────────────────────────────────────────────────────────────┤
│                   UI 基础组件层                             │
│  按钮/复选框/文本框 │ 滚动视图 │ 搜索栏 │ 单词卡片/抽认卡  │
│            多项选择题 │ 跨文字渲染引擎                      │
├─────────────────────────────────────────────────────────────┤
│                    业务逻辑层                               │
│  账户系统 │ 单词库引擎 │ 学习计划引擎 │ 搜索匹配 │ 进度追踪 │
├─────────────────────────────────────────────────────────────┤
│                      数据层                                 │
│  accounts.txt  words.txt  progress.txt  plans.txt           │
├─────────────────────────────────────────────────────────────┤
│                     核心框架                                │
│     全局状态管理器(AppState) │ 树形菜单导航 │ 主循环调度    │
└─────────────────────────────────────────────────────────────┘
```

**页面展示层**是用户直接交互的部分，包含 10 个功能页面。每个页面由独立的 `.c` 文件实现，通过函数指针注册到导航树中。页面函数遵循统一签名 `void show(void)`，每帧由主循环调用一次。

**UI 基础组件层**提供了一套即时模式 GUI 控件库，包含按钮、文本框、复选框、滚动视图、搜索栏、单词卡片、抽认卡（含翻转动画）和多项选择题等组件。这套组件库完全从零开发，不依赖任何第三方 UI 框架，底层仅调用 raylib 的 2D 绘制和文字渲染 API。

**业务逻辑层**封装了账户认证（djb2 哈希、会话管理）、单词库操作（CRUD、通配符搜索、进度追踪）和学习计划管理（每日目标、跨日自动推进）等核心功能。所有业务函数均为纯逻辑操作，不直接操作 UI 元素，保证了逻辑与表现的分离。

**数据层**使用四种管道分隔纯文本文件存储全部持久化数据。选择文本格式而非 SQLite 或 JSON 的原因有三：其一，纯文本可以用任何编辑器直接查看和修改，方便用户自定义单词库；其二，文本文件的读写不引入任何第三方库依赖；其三，管道分隔格式简单直观，解析代码仅需几十行，无需引入复杂的词法分析器。

**核心框架层**定义了 `AppState` 全局状态单例和 `MENU` 树形导航结构，是串联所有模块的中枢。主循环每帧先更新输入状态，再依次绘制顶栏、导航栏和当前页面，最后检查窗口关闭事件，形成典型的游戏引擎渲染循环。

### 1.2 系统启动流程

系统启动遵循严格的初始化顺序。每个初始化步骤都设有容错逻辑：如果文件不存在则创建默认数据，如果字体加载失败则回退到平台内置字体，确保系统在任何环境下都能正常启动。

```
程序启动
  │
  ├─ InitWindow(1600x1000, "背单词软件")
  │     创建图形窗口，设置为 60FPS 固定帧率
  │     窗口分辨率固定为 1600x1000，保证 UI 布局一致性
  │
  ├─ AppState_Init()
  │     全局状态单例归零，建立 account/plan 的依赖注入指针
  │     初始化 UI 主题为浅色模式
  │
  ├─ load_words_from_file("data/words.txt")
  │     逐行读取管道分隔文本，解析单词、音标、释义、例句字段
  │     构建 WordEntry 动态数组 g_wordLibrary[]
  │     文件缺失？→ 使用10个硬编码后备单词（确保 demo 可运行）
  │
  ├─ Account_Init()
  │     解析 accounts.txt，加载用户信息数组
  │     文件缺失？→ 创建 admin/admin 默认管理员账户
  │
  ├─ Plan_Init()
  │     解析 plans.txt，加载学习计划数组
  │     文件缺失？→ 添加一周入门/半月巩固/三十天进阶/六十天冲刺 4 个预设计划
  │
  ├─ init_words()
  │     将 g_wordLibrary 中的单词条目投影到 g_words[300] 固定数组
  │     每个单词附加 WordProgress 元数据（初始为全零）
  │     然后从 progress.txt 读取历史进度覆盖默认值
  │
  ├─ load_fonts()
  │     依次加载英文字体、IPA 音标字体、中文字体、合并字体
  │     每一步失败后自动尝试平台后备字体，确保至少有一种字体可用
  │
  ├─ InitMenuTree()
  │     构建 7 个顶级节点 + 5 个子节点的树形菜单
  │     每个节点绑定对应的页面渲染函数指针
  │
  ├─ RefreshReviewList()
  │     遍历 g_words[]，将未掌握的单词索引收集到待复习列表
  │     仅筛选 mastered == false 的单词，按计划剩余量截断
  │
  └─ 主循环 (每帧约60次):
        ├─ UIBegin()
        │     重置 hotItem/activeItem 状态，更新鼠标位置
        │     这是即时模式 GUI 的核心——每帧重新计算交互状态
        │
        ├─ BeginDrawing() / ClearBackground()
        │
        ├─ 绘制顶栏 + 用户按钮
        │     顶栏显示应用标题和当前登录用户
        │     点击用户按钮跳转到账号管理页面
        │
        ├─ 绘制左侧导航树
        │     宽度 250px，递归渲染菜单树节点
        │     当前活动节点高亮，鼠标悬停变色
        │
        ├─ CURRENT_MENU->show()
        │     通过函数指针多态调用当前活动页面的渲染函数
        │     这是整个 UI 的核心分发机制
        │
        ├─ 如需登录 → 绘制登录覆盖层
        │     半透明遮罩 + 居中登录表单，阻止访问其他功能
        │
        ├─ EndDrawing()
        │
        └─ UIEnd()
              │
              └─ 窗口关闭？→ 释放资源（字体、菜单树、单词库） → 程序退出
              └─ 否则 ──────────────────────────────────────→ 下一帧
```

初始化阶段的容错设计是系统"零配置部署"理念的具体体现。无论用户是否完整下载了 data 目录下的配套文件，系统都能够正常启动并展示一个可用的界面。对于缺失的字体文件，系统会自动降级到操作系统自带的字体；对于缺失的数据文件，系统会创建带默认数据的全新文件。这种设计使得系统作为 zip 压缩包分发成为可能——用户解压后直接双击即可运行，无需任何前置配置步骤。

### 1.3 模块依赖关系

下图展示了从入口文件 `main.c` 出发的完整依赖树。依赖方向是单向的：main.c 作为组装点，将各模块"装配"在一起；页面函数依赖 UI 组件库和业务逻辑模块，但业务逻辑模块绝不反向依赖 UI。这保证了视图层和逻辑层可以独立演进。

  ├── modules/account.c ───── 用户注册/登录/注销，密码 djb2 哈希
  ├── modules/plan.c ──────── 学习计划管理，跨日期自动推进
  ├── modules/fonts.c ─────── Unicode 级多语言文字渲染引擎

        └── ui/pages/
              ├── page_home.c           → raylib_word_ui.c + fonts.c
              ├── page_learn.c          → raylib_word_ui.c + fonts.c

              ├── page_plan.c           → raylib_word_ui.c + plan.c

从依赖树可以看出，`raylib_word_ui.c` 是被所有页面共同依赖的核心组件，承担了所有图形控件的渲染工作。`fonts.c` 被大多数页面依赖，负责混合文本的正确渲染。`plan.c` 被三个学习模式页面（卡片背单词、选词背单词、测试）共同依赖，说明学习计划与学习行为是紧耦合的——用户每次完成一个单词的学习后，计划模块会自动更新今日进度。`words.c` 被搜索页面和词库管理页面依赖，因为这两个页面需要读写单词库数据。

## 二、核心控制模块

如果说业务逻辑模块是系统的"器官"，那么核心控制模块就是系统的"骨骼"与"神经"。`app_state.c` 承载了全部运行时数据，`tree_menu.c` 定义了页面之间如何组织和跳转，二者共同构成了系统的骨架。

### 2.1 全局状态管理器（core/app_state.c）

在图形应用程序中，状态管理是架构设计中最关键也最容易失控的环节。传统的事件驱动 GUI 框架（如 Qt 的信号/槽、Android 的 Bundle/Intent）往往导致状态散落在各个控件对象中，跨页面共享时需要显式传递或序列化。本系统采用了一种更激进的方案：**全局单例状态容器**。所有可变运行时状态集中存储在一个 `AppState` 结构体中，各业务模块通过依赖注入方式持有指向其子状态的指针，实现低耦合数据共享。

```
AppState (全局单例 g_app, 约 2KB 内存占用)
  │
  ├── UIStyle style ────────── 主题配色(13色) + 字体句柄(4个) + 字号 + 间距
  ├── UIState uiState ──────── 每帧重算: 鼠标位置、按键状态、hotItem/activeItem/focusItem
  ├── bool isDarkMode ──────── 深色/浅色主题标记，切换时仅更新 UIStyle 指针
  ├── MENU* rootMenu ───────── 导航树根节点(永不改变)
  ├── MENU* currentMenu ────── 当前活动页面节点(随导航变化)
  ├── MenuStack menuStack ──── 导航历史栈(最大深度10层，支持返回)
  │
  ├── LearnState learn ──────── 浏览模式: 当前词索引、仅显示未掌握过滤、滚动偏移
  ├── ReviewState review ────── 抽认卡模式: 待复习单词列表、当前进度、会话统计
  ├── TestState test ────────── 测试模式: 题目数组、当前题号、正确数、完成标志
  ├── SelectWordState select ── 选词模式: 题目列表、当前题号、累计正确数
  ├── SearchState search ────── 搜索模式: 查询字符串、结果列表、滚动偏移
  │
  ├── AccountState account ──── 用户数组、当前用户索引、登录状态、登录消息
  ├── PlanState plan ────────── 计划数组、活跃计划索引、每日统计
  └── bool loginRequired ────── 是否强制登录(由设置页面控制)
```

乍看之下，将所有状态塞进一个全局结构体似乎违背了封装原则。但在即时模式 GUI 的场景下，这种设计带来了显著的好处：

**第一，状态与 UI 完全解耦。** 每个页面函数是纯函数——输入是 `AppState` 中的状态字段，输出是绘制调用。页面之间没有继承关系，不需要通过信号/槽通信。页面 A 修改了某个状态，页面 B 在下一帧渲染时自动感知变化。

**第二，避免了序列化开销。** 在保留模式 GUI 中，从一个对话框切换到另一个对话框，通常需要将当前状态序列化再反序列化。而在本系统中，"切换页面"仅仅是修改 `currentMenu` 指针，状态保持在原地不动。从"浏览学习"切换到"抽认卡"再切回来，浏览模式的分页位置、滚动偏移等状态完好无损。

**第三，依赖注入而非全局变量。** 尽管 `g_app` 在文件作用域内全局可见，但业务模块（account.c、plan.c）并不直接访问它。它们通过 `Account_SetState(&g_app.account)` 和 `Plan_SetState(&g_app.plan)` 获得指向自己子状态的指针，这使得这些模块可以被独立测试——传入不同的状态指针即可模拟不同场景。

### 2.2 树形菜单导航（core/tree_menu.c）

导航系统是用户与功能模块之间的桥梁。在一个拥有 10+ 页面的应用中，如何组织导航结构直接影响用户体验。本系统选择了**多路树**作为导航数据结构——每个节点可以拥有多个子节点，用户通过逐层展开来定位目标功能。

这种设计之所以比扁平 Tab 或抽屉式菜单更好，是因为它能够自然表达功能的层级关系。例如，"卡片背单词"、"选词背单词"、"测试"三种复习模式都隶属于"背单词"这一更抽象的概念，在树形结构中它们被组织为"背单词"节点的三个子节点，逻辑清晰，用户在脑中构建的导航地图与代码结构完全一致。

**菜单树结构：**

```
根节点 (Root)
  ├── 学单词 ─────────────── 浏览模式，逐词展示详细释义和例句
  ├── 背单词 ─────────────── 复习模式入口(不直接显示内容)
  │     ├── 卡片背单词 ───── 抽认卡翻转，认识/不认识判定
  │     ├── 选词背单词 ───── 看中文释义，从4个英文候选中选择
  │     └── 测试 ─────────── 看英文单词，从4个中文释义中选择
  ├── 查找单词 ───────────── 通配符搜索，实时匹配
  ├── 学习计划 ───────────── 计划入口
  │     ├── 计划管理 ─────── 创建、激活、删除学习计划
  │     └── 学习进度 ─────── 全部单词的掌握情况一览表
  ├── 设置 ───────────────── 深色/浅色主题切换、关于信息
  ├── 账号管理 ───────────── 注册、登录、注销
  └── 词库管理 ───────────── 单词库的增删改查，支持全文编辑
```

**导航栈操作流程：**

当用户从主页进入"卡片背单词"再返回时，导航栈记录着完整的路径信息，确保返回操作始终回到正确的页面。这种"压栈-弹栈"的模式与浏览器的前进/后退机制完全一致，用户心智负担为零。

```
用户点击 "卡片背单词"
  │
  ├─ StackPush(stack, currentMenu)    // 将"背单词入口"压入历史栈
  ├─ currentMenu = 卡片背单词节点      // 切换到抽认卡页面
  │
  └─ ...用户在抽认卡页面操作后点击 "返回"...
       │
       ├─ StackPop(stack)              // 从栈顶弹出
       └─ currentMenu = "背单词入口"   // 恢复到上一级页面
```

**核心数据结构：**

```c
typedef struct menu {
    void (*show)(void);          // 页面渲染函数指针 —— 实现多态导航
    struct menu* child[8];       // 子节点数组，最大出度 8
    struct menu* parent;         // 父节点指针，支持自底向上回溯
    int childindex;              // 已挂载子节点的数量(0~8)
} MENU;

typedef struct {
    MENU* menuStack[10];         // 最大深度 10 层的导航栈
    int Stacktop;                // 栈顶索引，-1 表示空栈
} MenuStack;
```

这里值得注意的设计细节：`show` 函数指针的运用实现了页面切换的"多态"——主循环不需要知道当前是哪个页面，只需调用 `CURRENT_MENU->show()` 即可。新增页面时，只需编写对应的 `MenuXxx_Show()` 函数并注册到菜单树上，主循环代码完全不需要修改。这本质上是对"开闭原则"的一种 C 语言实践。

## 三、业务逻辑模块

### 3.1 用户账户系统（modules/account.c）

英语学习是一个长周期的过程，通常持续数周甚至数月。在这个时间跨度内，用户可能使用多台设备，也可能与家人共享同一台电脑。因此，系统需要支持多用户账户，并确保每个用户的进度数据相互隔离。同时，考虑到这是一个本地桌面应用（无服务端），安全设计需要在可用性和保护强度之间取得平衡。

**初始化流程：**

```
程序启动
  │
  ├─ accounts.txt 存在？
  │     ├─ 是 → 逐行解析管道分隔字段，加载至 User 数组(最多50个用户)
  │     │       字段: username|passwordHash|createdTime|lastLoginTime|correct|total
  │     └─ 否 → 创建默认管理员账户 admin/admin
  │             写入 accounts.txt，确保首次启动即有可用账户
  │
  └─ 初始化完成，currentUserIndex = -1(未登录状态)
```

**登录验证流程：**

```
用户输入用户名 + 密码，点击登录
  │
  ├─ 遍历 User 数组按用户名匹配
  │     └─ 未找到 → 显示 "用户不存在"，等待重新输入
  │
  ├─ djb2(输入密码) 与存储的 passwordHash 字符串比对
  │     └─ 不匹配 → 显示 "密码错误"，等待重新输入
  │
  └─ 匹配成功
       ├─ currentUserIndex = 该用户的数组索引
       ├─ isLoggedIn = true
       ├─ 进度文件切换: progress.txt → progress_<username>.txt
       ├─ 计划文件切换: plans.txt → plans_<username>.txt
       ├─ 更新 lastLoginTime 为当前时间戳
       ├─ 刷新待复习单词列表(加载新用户的进度)
       └─ 持久化更新后的 accounts.txt
```

**多用户文件隔离机制：**

```
未登录状态:
  所有用户共享 → progress.txt          plans.txt

Alice 登录时:
  系统自动切换 → progress_alice.txt    plans_alice.txt
  (文件不存在则自动创建)

注销时:
  系统恢复 → progress.txt              plans.txt

Bob 登录时:
  系统自动切换 → progress_bob.txt      plans_bob.txt
  (与 Alice 的数据完全隔离，互不影响)
```

密码的存储采用 **djb2 哈希算法**。这个选择可能引发疑问：在 2025 年，为什么不使用 bcrypt、scrypt 或至少 SHA-256？答案是这是一个本地桌面应用，而非网络服务。密码哈希的目的不是防御服务器端数据库泄露后的暴力破解，而是防止使用同一台电脑的其他用户通过直接打开 `accounts.txt` 看到明文密码。在这种场景下，攻击者如果真的获得了文件系统的读取权限，他同样可以删除 `accounts.txt` 或修改可执行文件来绕过密码检查——换言之，更强的哈希算法并不能提供实质性的额外安全。djb2 的优点是实现仅需 5 行代码，无外部依赖，且生成的哈希值不会使 `accounts.txt` 的字段长度失控。

### 3.2 单词库引擎（modules/words.c）

单词数据是系统的核心资产。本系统支持用户自定义单词库——用户可以用文本编辑器直接编辑 `words.txt`，也可以通过程序内的词库管理页面进行增删改查。单词引擎需要处理两个核心问题：如何高效地在磁盘存储和内存表示之间同步数据，以及如何支持灵活的通配符搜索。

**数据的三层映射流程：**

设计上采用了"三层映射"方案来解耦存储、索引和运行时状态。每一层有明确的职责边界，层与层之间通过受控的转换函数连接。

```
第一层: words.txt (磁盘)
  格式: word|phonetic|definition|example|translation
  职责: 人类可读的持久化存储，支持记事本直接编辑
  举例: abandon|əˈbændən|放弃、遗弃|He abandoned his car in the snow.|他把汽车丢在雪地里。
  │
  ▼ load_words_from_file()  逐行解析管道分隔符，按行序构建动态数组
  │
第二层: g_wordLibrary[] (动态数组, 按文件顺序存储完整条目)
  类型: WordEntry* 指针数组
  职责: 保存单词的完整原始信息，作为所有上层操作的权威数据源
  特点: 长度可变(g_wordCount)，顺序与文件一致
  字段: char* word, *phonetic, *definition, *example, *exampleTranslation
  │
  ▼ init_words()  浅拷贝 WordEntry 指针 + 为每个单词创建零初始化进度跟踪结构
  │
第三层: g_words[300] (固定大小数组, 单词 + 进度元数据)
  类型: WordWithProgress 结构体数组
  职责: 运行时的高频访问载体，将单词本体与学习进度绑定
  特点: 长度固定(g_wordProgressCount ≤ 300)
  结构: { WordEntry entry; WordProgress progress; }
        其中 progress = { knownCount, unknownCount, lastReview, mastered }
  │
  ▼ load_progress() / save_progress()  按单词名匹配，更新进度字段 / 持久化到磁盘
  │
progress.txt (磁盘)
  格式: word|knownCount|unknownCount|lastReview
  职责: 记录每个单词的学习历史，独立于单词数据文件
```

三层映射的设计有一个重要特性：**单词数据和进度数据完全分离**。当用户编辑了 `words.txt`（例如新增了 50 个单词），重新加载后，已存在单词的 `knownCount` 和 `unknownCount` 会被保留（通过单词名匹配），新增的单词则从零进度开始。这使得用户可以随意扩充单词库而不丢失已有的学习成果。

**通配符搜索算法：**

系统自实现了一个轻量级的通配符匹配引擎，支持 `*`（零个或任意多个字符）、`?`（恰好一个字符）、`^`（锚定开头）、`$`（锚定结尾）四种元字符。这个引擎虽然无法与完整的正则表达式库媲美，但足以覆盖日常单词检索的全部场景，且代码体积不到 50 行。

```
用户输入搜索模式 (如 "ab*")
  │
  ├─ 模式含通配符字符 (^ $ * ?) ？
  │     ├─ 是 → search_words_by_regex()
  │     │       逐字符遍历模式串，按元字符语义匹配目标单词
  │     │       算法复杂度 O(m*n)，m=模式长度，n=单词长度
  │     │
  │     └─ 匹配数 > 0 → 返回通配符匹配的结果列表
  │            匹配数 = 0 → 自动降级到子串匹配（更宽松的搜索）
  │
  └─ 模式不含通配符 → search_words_simple()
        └─ 不区分大小写的子串匹配，遍历所有单词查找包含模式串的条目
```

**检索示例：**

| 输入模式 | 匹配逻辑 | 匹配结果示例 |
|----------|---------|------------|
| `ab*` | 开头为 "ab"，后续任意 | abandon, abstract, absurd |
| `*tion` | 结尾为 "tion"，前面任意 | education, information, nation |
| `h?t` | "h" + 一个任意字符 + "t" | hat, hit, hot (不会匹配 "heart") |
| `^test` | 严格以 "test" 开头 | test, testing, testify |
| `hello` | 无通配符 → 子串匹配 | hello, helloworld, prehello |

### 3.3 学习计划引擎（modules/plan.c）

许多背单词软件的共同问题是：用户打开后不知道今天该学多少，容易要么学得过少（进步缓慢），要么学得过多（产生倦怠）。学习计划引擎的目标是为用户提供可量化的每日学习目标和自动化进度追踪，让坚持变得可度量。

**跨日自动推进流程：**

计划引擎的核心智能在于"感知时间流逝"——它能够自动判断用户是否跨越了日期边界，并在新的一天自动推进计划进度，无需用户手动打卡。实现方式是将每条计划的 `lastStudyDate` 存储为 `yyyymmdd` 格式的整数（如 20260520 代表 2026 年 5 月 20 日），每次学习时与当前日期比较。

```
用户每完成一个单词的学习(点击"认识"或"不认识")
  │
  └─ Plan_AddStudiedToday(1)
       │
       └─ Plan_CheckNewDay()
            │
            ├─ 获取当前日期，编码为 yyyymmdd 整数
            │
            ├─ 与计划中存储的 lastStudyDate 比较
            │     ├─ 相等(同一天) → studiedToday += 1
            │     │     这是最常见的情况，用户正在今天的会话中学习
            │     │
            │     └─ 不等(跨天了) → currentDay++
            │                       studiedToday = 1 (这是新一天的第一个单词)
            │                       lastStudyDate = today
            │                       这个分支不需要用户任何操作，自动完成
            │
            └─ 将更新写回 plans.txt
                 │
                 ├─ studiedToday >= dailyWordCount → "今日目标已达成!"
                 │     用户仍可继续学习，但已完成当日计划
                 │
                 └─ studiedToday < dailyWordCount → "今日还需学习: N 个单词"
```

**四大预设学习方案：**

为了降低新用户的决策成本，系统内置了四个经过科学设计的默认计划，覆盖从入门到冲刺的不同强度需求。用户也可以创建完全自定义的计划，设置任意的每日单词量和总天数。

| 计划名称 | 每日单词量 | 总天数 | 累计词汇覆盖 | 适用人群 |
|---------|-----------|-------|------------|---------|
| 一周入门计划 | 10 词/天 | 7 天 | 70 词 | 零基础学习者，低强度适应 |
| 半月巩固计划 | 15 词/天 | 15 天 | 225 词 | 有一定基础，寻求系统化复习 |
| 三十天进阶计划 | 20 词/天 | 30 天 | 600 词 | 备考四级/六级的学生 |
| 六十天冲刺计划 | 30 词/天 | 60 天 | 1800 词 | 备考托福/雅思/GRE |

### 3.4 复习策略与进度追踪

系统的复习策略参考了**间隔重复（Spaced Repetition）**的核心理念——不是机械地按固定间隔重复所有单词，而是根据用户对每个单词的掌握程度动态调整。实现上做了简化：用一个计数器 `knownCount` 追踪用户认识该单词的累计次数，当连续认识 3 次（`MASTERED_THRESHOLD`）后标记为"已掌握"，该单词从复习池中移除。

```
RefreshReviewList() 每轮复习开始时调用
  │
  ├─ 遍历 g_words[0..g_wordProgressCount-1]
  │     └─ 筛选条件: mastered == false (尚未达到掌握阈值)
  │     └─ 收集符合条件的单词索引 → reviewWords[]
  │     └─ 如果活跃计划存在，截断数量不超过今日剩余量
  │
  ├─ reviewWords 为空 → 显示 "恭喜！所有单词已掌握!"
  │     用户可以通过"清除进度"功能重置所有单词状态
  │
  └─ reviewWords 非空 → 逐个展示抽认卡
       │
       ├─ [正面] 显示英文单词 + 音标
       │
       ├─ 用户点击翻转 → [背面] 显示中文释义 + 例句 + 例句翻译
       │     翻转动画使用 cos(t*PI) 缩放 X 轴实现 3D 视觉效果
       │
       ├─ 用户判断:
       │     ├─ 点击 "认识" → knownCount++
       │     │     └─ knownCount >= 3 ? → mastered = true (从复习池移除)
       │     │
       │     └─ 点击 "不认识" → unknownCount++
       │           └─ 保留在复习池中，下次复习时继续出现
       │
       ├─ 更新 lastReview 时间戳
       │
       └─ 全部单词复习完毕 → 显示本轮统计(认识N个，不认识M个)

掌握度可视化:
  掌握率 = knownCount / (knownCount + unknownCount)
    ├─ ≥ 80% → 绿色 ● (已掌握——可以移出复习池)
    ├─ 50% ~ 79% → 橙色 ● (学习中——需要继续巩固)
    └─ < 50% → 红色 ● (薄弱——需要重点突破)
```

这个策略比简单的艾宾浩斯遗忘曲线更易于实现，因为不需要为每个单词存储复杂的间隔时间表。它的核心假设是：如果一个单词连续 3 次出现在复习中都被用户认识，说明用户已经基本掌握（可能在 3 天内被测试了 3 次，也可能在 1 周内被测试了 3 次——这取决于复习池的大小和用户的复习频率）。虽然不如 SuperMemo 的 SM-2 算法精确，但对于桌面端非联网应用来说，这种简单方案反而更可预测、更透明。

## 四、UI 基础组件层

### 4.1 即时模式 GUI 系统（ui/raylib_word_ui.c）

本系统选择了一个相对小众的 UI 开发范式——**即时模式（Immediate Mode）GUI**。在这个范式中，UI 不是静态的对象树，而是每一帧从头执行的一系列过程调用。一个按钮的"创建"、"事件处理"和"渲染"在同一个 `UIButton()` 函数调用中完成，函数返回 `true` 表示本帧该按钮被点击。

**两种范式对比：**

```
保留模式 (Qt / GTK / WinForms):
  CreateWidget()   → 创建按钮对象，分配内存
  SetProperties()  → 设置位置、大小、文字、样式
  ConnectSignal()  → 绑定点击回调函数
  EventLoop()      → 事件循环：等待用户操作，触发回调
  特点: 控件对象常驻内存，框架管理生命周期

即时模式 (Dear ImGui / 本系统):
  每帧调用 UIButton() → {
      检测鼠标是否在按钮矩形内 → 更新 hotItem/activeItem
      根据状态选择颜色和样式 → 调用 raylib 绘制函数
      返回是否在本帧被点击
  }
  特点: 无持久对象，无回调，状态完全显式
```

选择即时模式而非引入 Qt 或 GTK，是基于实际需求的权衡。引入 Qt 意味着安装包需要携带约 30-50MB 的动态库依赖，且需要维护 C++/Qt 的绑定代码；引入 GTK 则意味着在 Windows 上的部署体验会非常糟糕（需要安装 MSYS2 运行时环境）。而 raylib 作为单头文件 C 库，静态链接后整个可执行文件仅约 2-5MB，这完全符合"零配置部署"的核心理念。即时模式 GUI 的额外好处是代码极其精简——整个 UI 组件库（约 2500 行 C 代码）实现了 10+ 种控件，而在 Qt 中这可能意味着数十个类文件和 MOC 预处理。

**热点状态机的运行机制：**

即时模式 GUI 的一个关键挑战是状态管理——在没有任何对象存储状态的情况下，如何区分"按钮刚被点击"和"按钮持续被按住"？答案是使用一个全局的状态跟踪系统，通过 `hotItem` 和 `activeItem` 两个整数变量来模拟控件状态。

```
每帧开始: hotItem = 0, activeItem = 0 (重置所有状态)
  │
  ├─ 遍历每个控件:
  │     │
  │     ├─ 鼠标在控件区域内?
  │     │     ├─ 否 → 跳过，控件的绘制使用默认样式
  │     │     │
  │     │     └─ 是 → hotItem = 该控件ID (标记为"被悬停")
  │     │           ├─ activeItem == 0（无其他控件处于激活态）
  │     │           │     → 应用悬停样式（颜色变亮或加边框）
  │     │           │
  │     │           ├─ 鼠标左键刚按下（!mouseDown 且 !mousePressed）
  │     │           │     → activeItem = 该控件ID (标记为"被激活")
  │     │           │     → 应用按下样式（颜色变暗或下沉效果）
  │     │           │
  │     │           └─ 鼠标左键松开（mouseDown 且 mousePressed）
  │     │                 └─ activeItem == 该控件ID → 触发点击！返回 true
  │     │
  │     └─ activeItem 的生命周期:
  │           鼠标按下时设置 → 鼠标松开时清除
  │           如果在控件外部松开 → activeItem 保留但失去热区，不会触发点击
  │
  └─ 下一帧重新开始，hotItem 和 activeItem 被 UIBegin() 重置
```

这个状态机的巧妙之处在于：它仅使用两个全局变量就模拟了整个 UI 系统的交互状态，无需任何对象分配。所有控件共享同一套状态变量，但每个控件有不同的 ID，通过比较 ID 来判断"这个操作是发生在自己身上还是在其他控件上"。

**主题系统：**

系统支持深色和浅色两种视觉主题，通过 `AppState_ToggleDarkMode()` 一键切换。主题数据是静态定义的 13 色结构体，切换时仅修改 `UIStyle` 中的指针，无需重新创建任何 UI 控件。这保证了主题切换的瞬时响应。

```
浅色主题 (Light)                            深色主题 (Dark)
  背景基色:  #F5F5F5 (奶白)                  背景基色:  #1E1E2E (深蓝黑)
  卡片/面板: #FFFFFF (纯白)                  卡片/面板: #2D2D3F (深灰蓝)
  主文字色:  #333333 (深灰)                  主文字色:  #E0E0E0 (浅灰)
  主色调:    #4A90D9 (柔和蓝)                主色调:    #6CB4EE (明亮蓝)
  成功色:    #67C23A (翠绿)                  成功色:    #7ED321 (亮绿)
  危险色:    #F56C6C (柔和红)                危险色:    #FF6B6B (亮红)
  边框色:    #E0E0E0                          边框色:    #3D3D5C
  悬停色:    #ECF5FF (极浅蓝)                悬停色:    #3A3A5C
  ←─────────── ToggleDarkMode() ───────────→
```

### 4.2 多语言文字渲染引擎（modules/fonts.c）

raylib 的原生 `DrawTextEx()` 函数有一个限制：一个调用只能使用一种字体。这意味着如果你用中文字体渲染一段包含英文和中文的文本，英文字符会以中文字体的英文字形显示，通常效果较差（间距不协调、基线不对齐）。反之，用英文字体渲染包含中文的文本，中文字形会缺失，显示为空白方块。

本系统的解决方案是一个逐字符分析 Unicode 码位并动态切换字体的渲染函数 `DrawTextAuto()`。这个函数的实现不到 100 行，却解决了中英混排场景下的核心痛点。

```
输入文本: "abandon /əˈbændən/ 放弃"

逐字符遍历渲染:
  │
  ├─ 'a','b','a','n','d','o','n',' '
  │     └─ 码位 U+0061~U+006E (Basic Latin, ASCII) → 英文字体渲染
  │        字体: DejaVuSans.ttf (专门优化的西文字形)
  │
  ├─ '/','ə','ˈ','b','æ','n','d','ə','n','/'
  │     └─ 码位 U+0259(ə)、U+00E6(æ) → IPA/拉丁扩展字体渲染
  │        字体: LatinFont (包含完整的 IPA 音标符号)
  │
  ├─ ' '
  │
  └─ '放','弃'
        └─ 码位 U+653E、U+5F03 (CJK Unified Ideographs) → 中文合并字体渲染
           字体: MergedFont (NotoSansCJK + raylib 默认字体回退)
```

合并字体（MergedFont）的构建策略也很巧妙。它不是简单地使用中文字体渲染一切，而是通过 raylib 的字形合并 API 将默认字体中的 ASCII 字形、IPA 扩展字形和中文字体中的 CJK 字形合并到一个字体对象中。这样，当需要渲染中文字符时使用这个合并字体，ASCII 字符也能获得较好的显示效果，不需要逐字符切换到英文字体。这在不方便逐字符分析（如 raylib 内置的 `DrawTextRec()` 自动换行）的场景下特别有用。

**字体后备链：**

```
英文字体加载顺序:
  1. ./data/fonts/DejaVuSans.ttf        (项目自带，跨平台一致)
       ↓ 加载失败(文件缺失)
  2. Windows: C:/Windows/Fonts/arial.ttf (系统预装)
  3. macOS: /System/Library/Fonts/Helvetica.ttc
  4. Linux: /usr/share/fonts/truetype/dejavu/DejaVuSans.ttf
       ↓ 全部失败
  5. GetFontDefault()                    (raylib 内置 8x8 位图字体)
       永远成功，但字形质量较低

中文字体加载顺序:
  1. ./data/fonts/NotoSansCJK.otf       (项目自带)
       ↓ 加载失败
  2. 各平台字体目录搜索中文字体(宋体/黑体/Noto Sans CJK...)
       ↓ 全部失败
  3. GetFontDefault()                    (仅支持 ASCII，中文显示为方块)
```

### 4.3 抽认卡翻转动画

卡片背单词页面的核心交互是"抽认卡翻转"——用户先看到英文单词（正面），点击后卡片翻转显示中文释义（背面），然后用户判断自己是否认识。这个翻转动画通过 `UIFlashCard()` 函数中的三维仿射变换实现，利用 `cos()` 函数模拟卡片绕 Y 轴旋转的视觉效果。

```
每帧调用 UIFlashCard():
  │
  ├─ 根据动画进度计算翻转参数 t
  │     t 从 0 (完全正面) 到 1.0 (完全背面)
  │     使用 Lerp(指数平滑插值) 实现缓入缓出，而非线性过渡
  │
  ├─ t < 0.5 (显示正面——英文单词):
  │     scaleX = cos(t * PI)
  │     当 t=0:   scaleX=1.0 (卡片完全展开，英文单词清晰可见)
  │     当 t=0.5: scaleX=0.0 (卡片翻转到侧面，完全不可见)
  │     视觉效果: 卡片从水平逐渐"收缩"到一条线
  │
  ├─ t >= 0.5 (显示背面——中文释义 + 音标):
  │     scaleX = cos(t * PI)
  │     当 t=0.5: scaleX=0.0 (从一条线开始展开)
  │     当 t=1.0: scaleX=-1.0 (完全翻转到背面，镜像显示)
  │     视觉效果: 卡片从一条线"展开"到完全水平，文字已变为镜像
  │
  ├─ DrawTextPro(..., scaleX, 1.0, origin)
  │     以卡片中心点为锚点进行 X 轴缩放
  │     当 scaleX 为负时，文字自然呈现镜像效果(背面内容)
  │     raylib 的 DrawTextPro 原生支持负 scale 值
  │
  └─ 卡片底部:
        ├─ 正面时: 显示"点击翻转查看释义"提示文字
        └─ 背面时: 显示"认识 ✓"和"不认识 ✗"两个按钮
             └─ 点击任一按钮 → t 重置为 0，载入下一个待复习单词
```

动画使用的是简单的三角函数而非物理引擎或关键帧系统。`cos(t * PI)` 在 t=[0, 1] 区间内从 1 平滑过渡到 -1，恰好模拟了卡片翻转 180 度的效果。正面的文字在翻转过程中逐渐变窄（模拟透视收缩），翻过中线后从镜像方向逐渐展开（模拟背面的透视效果）。这个简单的数学技巧远比重写一个 3D 渲染管线省力，但在用户看来视觉体验与真正的 3D 翻转并无明显差异。

## 六、数据持久化方案

### 6.1 文件存储设计

系统的数据持久化选择了**管道分隔纯文本**而非 SQLite 或 JSON/XML。这是在"零依赖"原则下的自然选择。SQLite 需要链接 `libsqlite3`，JSON/XML 需要引入解析库（或手写解析器，但代码量远大于管道分隔解析）。管道分隔格式的解析代码仅需 `strtok` + `sscanf` 几十行，且生成的文本文件对人类肉眼友好，用户可以直接用记事本编辑。

| 文件 | 字段结构 | 编码 | 最大大小 |
|------|---------|------|---------|
| `words.txt` | `word\|phonetic\|definition\|example\|translation` | UTF-8 | ~10KB (150词) |
| `accounts.txt` | `username\|passwordHash\|createdTime\|lastLoginTime\|correct\|total` | UTF-8 | ~2KB (50用户) |
| `progress.txt` | `word\|knownCount\|unknownCount\|lastReview` | UTF-8 | ~5KB (150词) |
| `plans.txt` | `name\|dailyWordCount\|totalDays\|currentDay\|createdAt\|lastStudyDate\|studiedToday\|isActive` | UTF-8 | ~1KB (20计划) |

**示例数据 (words.txt):**

```
abandon|əˈbændən|放弃、遗弃|He abandoned his car in the snow.|他把汽车丢在雪地里。
abstract|ˈæbstrækt|抽象的、摘要|The idea is too abstract to understand.|这个想法太抽象了，难以理解。
```

### 6.2 多用户数据隔离方案

多用户隔离的设计思路是"文件路径重定向"而非"单文件多表"。每个用户拥有独立的 `progress_<username>.txt` 和 `plans_<username>.txt` 文件，文件名即表示了所有权。这种方案的优点显而易见：

第一，**数据天然隔离**——不可能出现"读错了用户的进度"这种 bug，因为文件名本身就防止了混淆。

第二，**备份和迁移简单**——如果你想把自己的学习进度迁移到另一台电脑，只需要复制 `progress_alice.txt` 和 `plans_alice.txt` 两个文件。

第三，**没有锁定/并发问题**——因为是本地单用户应用，不涉及多进程同时读写同一个文件。

```
未登录状态:
  progress.txt  plans.txt  (全局共享的默认文件)
       │            │
       │ Alice 登录时:
       │   set_progress_file_path("progress_alice.txt")
       │   Plan_SetFilePath("plans_alice.txt")
       │   所有后续读写自动定向到 alice 的文件
       │            │
       ├──→ progress_alice.txt  plans_alice.txt
       │
       │ Alice 注销时:
       │   set_progress_file_path("progress.txt")
       │   Plan_SetFilePath("plans.txt")
       │   恢复默认文件路径
       │            │
       ├──→ progress.txt  plans.txt
       │
       └── Bob 登录时:
            路径重定向至 progress_bob.txt / plans_bob.txt
```

## 七、跨平台构建与发布

### 7.1 构建系统设计

系统的构建基于 CMake 3.10+，支持 Linux（GCC/Clang）、Windows（MSVC 和 MinGW-w64）以及 macOS（Apple Clang）。构建配置的核心挑战在于 raylib 的链接方式——静态链接和动态链接的处理逻辑完全不同。

```
src/*.c (21个源文件, ~8000行代码)
  │
  ▼
CMakeLists.txt
  编译标准: C11
  编译选项: -finput-charset=UTF-8 -fexec-charset=UTF-8 (确保中文常量正确编码)
  目标: main_c (可执行文件)
  │
  ├── Linux/macOS → GCC/Clang → 静态链接优先(.a)
  │     链接依赖: m pthread dl X11 Xrandr Xinerama Xcursor Xi Xxf86vm GL
  │     产物: 单体文件 ~2-5MB，无外部依赖
  │
  ├── Windows MSVC → 静态链接优先
  │     链接依赖: opengl32 gdi32 winmm
  │
  └── Windows MinGW-w64 (MSYS2 UCRT64) → 动态链接(.dll.a)
        链接依赖: glfw3 opengl32 gdi32 winmm
        DLL 运行时: raylib.dll, glfw3.dll (需与 exe 一同分发)
```

### 7.2 CI/CD 自动发布管道

每当推送一个以 `v` 开头的 Git 标签（如 `v5.0.3`），GitHub Actions 会触发一个三阶段的构建流水线。Linux 和 Windows 两个构建作业并行执行，最后在 Release 作业中汇合：

```
推送 v* 标签 (例如 v5.0.3)
  │
  ├── Linux Job (ubuntu-24.04) ──────────────────────┐
  │     ├─ apt 安装: cmake ninja-build fuse 等         │
  │     ├─ 从 GitHub 源码编译 raylib 6.0 (静态链接)    │
  │     │     关闭 BUILD_SHARED_LIBS 和 BUILD_EXAMPLES │
  │     ├─ Ninja 编译项目(Release 模式)                │
  │     ├─ ldd 验证: 无 libraylib.so 依赖(确保静态)    │
  │     ├─ 打包 tar.gz 便携版: main_c + data/          │
  │     └─ 打包 AppImage: AppDir + AppRun + desktop    │
  │                                                   │
  ├── Windows Job (windows-2025 + MSYS2 UCRT64) ─────┤
  │     ├─ MSYS2 安装: gcc cmake ninja raylib zip      │
  │     ├─ Ninja 编译项目                              │
  │     ├─ 复制运行库: raylib.dll glfw3.dll            │
  │     └─ 打包 zip 便携版: exe + dll + data/          │
  │                                                   │
  └── Release Job (ubuntu-24.04) ────────────────────┘
        ├─ 下载 linux-package 和 windows-package 两个 artifact
        ├─ 整理发布文件到 release/ 目录
        └─ softprops/action-gh-release@v2 创建 GitHub Release
              ├─ 背单词软件-x86_64.AppImage     (~5MB)
              ├─ 背单词软件-linux-x86_64.tar.gz (~2MB)
              └─ 背单词软件-windows-x86_64.zip  (~3MB)
```

Linux 端选择了从源码编译 raylib 而非使用 apt 的预编译包，原因是 apt 的 `libraylib-dev` 可能不是最新版本，且默认编译选项是动态链接。从源码编译可以精确控制 `BUILD_SHARED_LIBS=OFF`（强制静态链接），以及关闭无关的示例编译。最终生成的 AppImage 包含了全部依赖（raylib 静态链接 + 数据文件 + AppRun 启动脚本），用户下载后赋予执行权限即可运行。

## 八、关键技术决策与创新点

### 8.1 设计决策权衡

软件工程中不存在完美的设计方案，每个选择都伴随着代价。下表总结了本系统做出的五项核心决策及其理由和代价：

| 决策维度 | 选择方案 | 选择理由 | 代价 |
|---------|---------|---------|------|
| GUI 范式 | 即时模式 | 无对象生命周期管理；代码量约少 60%；状态完全显式可追踪 | 每帧重绘全部控件，CPU 消耗高于保留模式；长文本需每帧重新测量 |
| 数据存储 | 纯文本管道分隔 | 零外部库依赖；用户可用记事本直接编辑；Git 友好可 diff | 无法支持并发写操作；大数据量(>1000词)下无索引，遍历性能下降 |
| 编程语言 | C11 | 编译后二进制极小(静态链接 <5MB)；与 raylib C API 无缝对接 | 无 STL；需手工管理动态数组和字符串；缺少现代语言的包管理 |
| 密码安全 | djb2 哈希 | 无任何加密库依赖；有效防止明文密码被旁人直接看到 | 非密码学安全级别；不适用于需要抵抗离线暴力破解的网络传输场景 |
| 字体渲染 | 自研 DrawTextAuto | 完美解决 raylib 单字体限制下的中英混排问题；不引入 harfbuzz/pango 等重型库 | 维护成本集中在 fonts.c；每种新平台可能需要调整字体后备路径 |

### 8.2 创新亮点

**1. 即时模式 GUI + 双主题系统**

在未使用 Qt/GTK 等重型 GUI 框架的前提下，从零构建了一套完整的即时模式图形界面组件库，涵盖按钮、文本框（支持 UTF-8 中英文输入和光标操作）、复选框、滚动视图、搜索栏、单词信息卡片、抽认卡（含 3D 翻转动画）和多项选择题等 10 余种控件。控件支持悬停/按下/激活三种视觉状态和深浅双主题色彩方案，全部控件逻辑集中在约 2500 行 C 代码中。主题切换仅需修改一个结构体指针，无需重建任何界面元素。

**2. Unicode 码位级字体动态切换**

raylib 原生 API 仅允许使用单一字体渲染文本。本系统设计了 `DrawTextAuto()` 函数，逐字符分析 Unicode 码位范围（Basic Latin / IPA Extensions / CJK Unified Ideographs），自动在英文字体、IPA 音标字体和中文合并字体之间切换，解决了中英混排段落中英文字形不协调、音标符号缺失、中文字符显示为方块等长期困扰 C 语言图形应用的问题。该方案无需引入 harfbuzz、pango 等重量级文字排版库。

**3. 文件路径重定向实现多用户数据隔离**

摒弃了"单文件多表 + username 字段"的传统数据库设计思路，采用了"文件名即所有权"的轻量方案。每个用户的进度和计划数据存储在独立命名的文本文件中，登录和注销时仅做文件路径指针的重定向，无需遍历或过滤任何数据表。这种设计不仅消除了"读错用户数据"类 bug 的可能性，还使得用户数据的备份、迁移和重置变得极为简单——只需复制或删除对应文件即可。

**4. 自适应日期感知的学习计划引擎**

计划引擎通过将日期编码为 `yyyymmdd` 整数并在每次学习时比对，实现了对日期变化的自动感知。用户无需执行任何"签到"或"打卡"操作——连续使用 N 天，计划自动推进 N 天；如果中断了几天再回来，计划会自动跳到新的一天。这种"无感追踪"的设计降低了用户的认知负担，让坚持学习的阻力降到最小。

---

## 九、总结

本系统以 C11 语言配合 raylib 多媒体图形库，实现了一款功能完备的跨平台英语单词记忆桌面软件。系统涵盖用户管理、单词库 CRUD、四种差异化学习模式（浏览/抽认卡/选词/测试）、自适应学习计划管理、通配符搜索、学习进度追踪和深浅主题切换等核心功能模块。通过即时模式 GUI 范式与 Unicode 级字体切换渲染引擎，在约 8000 行 C 代码的轻量级技术栈上实现了良好的跨平台用户体验。

在架构层面，全局单例状态管理（AppState）+ 函数指针驱动的树形菜单导航构成了系统的"骨架"，使得 10 个独立页面能够无缝协作同时又保持代码层面的低耦合。在数据层面，管道分隔纯文本文件方案确保了系统的零依赖部署与高可维护性——用户无需安装任何数据库引擎、运行时环境或第三方库。在工程层面，项目通过 CMake 管理跨平台编译，通过 GitHub Actions 实现了 Linux AppImage 与 Windows zip 便携版的自动化构建与发布，具备从代码提交到用户交付的完整工程落地能力。

## 附录 B：编译与运行

./build.sh package    # 一键安装依赖 + 编译 raylib + 构建项目 + 打包

**Windows (MSYS2 UCRT64)：**
```bash
pacman -S mingw-w64-ucrt-x86_64-gcc mingw-w64-ucrt-x86_64-cmake \
          mingw-w64-ucrt-x86_64-ninja mingw-w64-ucrt-x86_64-raylib
mkdir build && cd build
cmake .. -G Ninja -DCMAKE_BUILD_TYPE=Release
ninja
# main_c.exe 位于 build/ 目录下
```

**macOS：**
```bash
brew install raylib cmake ninja
mkdir build && cd build
cmake .. -G Ninja -DCMAKE_BUILD_TYPE=Release
ninja
```

**跨平台自动构建（CI/CD）：**
```bash
git tag v5.0.4
git push origin v5.0.4
# GitHub Actions 自动触发，生成 AppImage + tar.gz + zip 三种分发包
```
